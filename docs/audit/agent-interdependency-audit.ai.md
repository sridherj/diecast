## Section 0 — Methodology

**Goal:** classify every cross-reference inside the harvested `cast-*` fleet as
**green** (target exists, naming consistent), **yellow** (target exists but the
invocation pattern is inconsistent vs. other call sites), or **red** (target missing,
legacy `taskos-*` survivor, or excluded SJ-personal placeholder).

**Dual-mode parser.** The audit recognises two flavors of agent source:

- **Markdown prompt** (default): when an agent dir contains `cast-<name>/cast-<name>.md`,
  the audit walks that file with the regex extractors below.
- **Python script** (Issue #4 / D4): when no `.md` prompt exists (today: `cast-preso-review`),
  the audit walks the dir's `.py` files via `ast.parse()` and scans each `ast.Constant`
  string literal with the same extractors. `.venv/` and `tests/` subtrees are excluded.

**Skills** are always markdown-flavored (one `SKILL.md` per skill dir).

**Reference extractors:**

- `(?<![A-Za-z0-9])/cast-[a-z][a-z0-9-]*` — slash-command invocation (`/cast-foo`).
- `(?<![A-Za-z0-9/_-])(cast-[a-z][a-z0-9-]+)(\s+(agent|skill))?` — bare prose mention.
- `\btaskos-[a-z][a-z0-9-]*` (case-insensitive) — Section 1 safety net for surviving
  legacy prefixes.
- Skill-tool invocations in markdown today read as bare references; the Diecast fleet
  does not yet emit `Skill(...)` tool-call literals so a separate AST matcher is not
  required for that pattern (caught by the prose matcher above).

**Single-pass-per-file performance contract** (Issue #11 / D11): each agent / skill
file is opened, parsed, and turned into an in-memory ref list exactly once;
`--mode=all` then runs the per-mode classifiers against that one parse. The cost of
`--mode=all` matches the cost of any single mode plus a constant.

**Classification rules (naming):**

- target is in the discovered fleet → **green** (or `self-reference` for
  intra-agent mentions).
- target is a flagged SJ-personal placeholder (`cast-startup-discover`,
  `cast-startup-enrich`, `cast-linkedin-ai`, `cast-reachout-packet`,
  `cast-opportunity-finder`, `cast-generate-news-digest`, `cast-google-alert-setup`,
  `cast-startup-pipeline`) → **red** (`references excluded SJ-personal agent`).
- target name still carries the legacy `taskos-` prefix → **red**
  (`legacy taskos-* prefix survived into diecast`).
- target is none of the above and is not in the fleet → **red** (`target not found
  in fleet`).

**Classification rules (cross-skill):**

- only edges where the target is a **skill** (kind == `skill` in the graph) are
  considered for Section 4.
- shared-skill targets (`cast-child-delegation`, `cast-interactive-questions`,
  `cast-agent-design-guide`) emit **yellow** when more than one invocation pattern is
  observed across the fleet (e.g., a mix of `/cast-child-delegation` slash commands and
  bare prose `cast-child-delegation`). A `comparison_site` example is attached so 2.4
  can pick a canonical form.
- everything else is **green**.

not caught by a literal scan. Mitigation: 2.4's dry-run pass executes each agent end-to-
end and surfaces these at runtime.
