# Web Research: Living Source of Truth — Round-Trip Write-Back, Provenance, Notification & Conflict Surfacing (Step 7)

**Goal context:** Refine Requirements v2 — Workflow-Aware, HTML-First Requirements Refinement.
Step 7 designs the round-trip mechanism by which downstream phases (exploration, planning, execution)
write requirement-affecting changes *back* into the requirements artifact — with **write-back as
additions/annotations, not silent rewrites** (FR-018, US7 S1); **provenance** of which phase/agent
originated it, surfaced in the version change summary (FR-020, US7 S3); **user notification** of what
changed and from where (FR-019, US7 S2); and **conflict surfacing** when a downstream change contradicts
an existing requirement — raised to the user, never silently overwritten (US7 S4). Downstream **agents**
are the typical write-back source (FR-013). Validated by tracing one downstream change end-to-end into
the files + notification (SC-006).

**Date:** 2026-06-11
**Researcher angle:** cast-web-researcher (7 expert lenses)
**Hard dependencies consumed:**
- Step 2 canonical-store note (`02-canonical-source-of-truth*.ai.md`) — **stable element surrogates +
  immutable version snapshots**. These are the *write target* and the *conflict unit*. Step 7 cannot
  detect conflicts or attach provenance without them.
- Step 4 annotation/iteration note (`04-annotation-and-iteration.ai.md`) — the **structured element-diff
  change-summary engine**, the **same-door comment API** (`author_type∈{human,agent}`), and the
  **append-only `comment_events` trail**. Step 4 *explicitly hands this machinery to Step 7* (its §Lens 5
  and open-question #4). Step 7 is largely a *composition* of these, not a greenfield build.

## TL;DR — The Verdict (for the synthesizer)

1. **Round-trip is "propose + notify + gate," NOT "auto-sync." This is the single most important framing.**
   The naive read of "living source of truth" is *automatic two-way synchronization* — downstream
   discoveries silently flow back into the requirements. **Reject that.** A silent auto-overwrite of a
   human-authored requirement is untraceable data loss (the spec calls it out: US7 S4 "never silently
   overwriting"). The correct model is the one mature tools converged on: a downstream change arrives as a
   **proposed change / suggestion** (Google-Docs *suggesting mode* / GitHub *PR*), is **attributed**
   (provenance), **notifies** the owner, and is **applied through a gate** — automatically only for
   low-blast-radius *additions*, never for *modifications* to existing human-authored elements.

2. **Model the write-back as a first-class `change_request` entity carrying `origin` + `base`, not a
   direct edit.** A downstream phase does not `UPDATE` an element; it `POST`s a proposed change anchored
   to Step 2's element surrogate, tagged with **where it came from** (origin phase/agent/artifact) and
   **what it was derived from** (the `base_version_id` it assumed). `origin` powers notification + audit;
   `base` powers conflict detection. The naive design forgets both. (§Lens 5.)

3. **Provenance = W3C PROV's three nuclei: Entity, Activity, Agent — mapped 1:1 onto FR-020.** The
   requirement revision is an *Entity*; the downstream phase run is an *Activity*; the human/AI is an
   *Agent*. PROV's three core relations are exactly what FR-020 needs: `wasGeneratedBy` (this revision was
   generated by the planning activity), `wasAttributedTo` (attributed to `cast-high-level-planner`),
   `wasDerivedFrom` (derived from `plan.collab.md` / the prior requirement). PROV's URI-per-version scheme
   is described by W3C as *"particularly suited for expressing provenance of documents."* Borrow the
   *shape*, not the full JSON-LD ceremony (same posture Step 4 took with W3C Web Annotation). (§Lens 4.)

4. **Conflict detection = three-way merge against a common base, i.e. optimistic concurrency.** Because
   every element carries a version identity (Step 2) and every `change_request` carries the `base_version_id`
   it was computed against, conflict is a deterministic predicate: *did the element diverge from `base`
   since the change was authored?* If unchanged → clean apply. If a human edited it since `base` AND the
   downstream change modifies it → `conflicted` → surface, never overwrite (US7 S4). This is textbook
   `conflictDetection: VERSION` (AWS AppSync) / git 3-way merge. **No CRDT/OT needed — co-editing is out
   of scope** (Step 4 already killed that premise). (§Lens 1, §Lens 5.)

5. **Notification = transactional outbox (reliability) + W3C Linked Data Notifications (surface).** The
   write-back-and-notify is a *dual write* — you must not commit the requirement change but lose the
   notification (or vice versa). Solve it with the **transactional outbox pattern**: write the change AND
   the notification event in the *same DB transaction*, then a relay delivers (at-least-once → idempotent
   notifications). The user-facing surface is naturally **W3C LDN** — the notification companion to Step 4's
   Web Annotation Protocol: each spec advertises an *Inbox*; downstream senders POST JSON-LD notifications;
   the human UI and other agents are consumers. Same-door, decentralized, agent-native. (§Lens 2, §Lens 4.)

6. **Step 7 reuses Step 4's change-summary engine verbatim for FR-020.** A write-back appears in the next
   version's change summary as a provenance-tagged delta: *"+FR-021 — added by planning / agent
   `cast-high-level-planner`, derived from `plan.collab.md`."* This is Step 4's structured element diff
   with a provenance badge — no new diff machinery. The append-only trail (Step 4's `comment_events`
   pattern, generalized to `change_request_events`) **is** the audit log — event-sourcing's core insight:
   *the event store IS the audit trail* (who/what/when/why, reconstructable). (§Lens 3, §Lens 5, §Lens 7.)

## Lens 1 — Expert Practitioner: how requirements tools actually round-trip downstream change (and where it hurts)

**The discipline that has fought this exact battle for 30 years is requirements traceability — and its
hard-won verdict is: propagate change as a *flagged proposal a human clears*, never as a silent edit.**
Bidirectional traceability (ISO 16404:2020) is defined as a two-way link where each entity knows the
other, supporting both *top-down impact analysis* and *bottom-up coverage analysis*
([Jama: Bidirectional Traceability](https://www.jamasoftware.com/requirements-management-guide/requirements-traceability/bidirectional-traceability/)).
**Step 7 is the bottom-up leg**: a downstream discovery (a plan, an execution finding) must flow *up* into
the requirement. The naïve failure mode the industry names explicitly: *"when an upstream requirement
changes, nothing automatically marks downstream items as potentially out of date… the typical result is
stale verification evidence that looks valid in the RTM but no longer matches the current requirement"*
([Jama: AI in Requirements Management](https://www.jamasoftware.com/blog/ai-requirements-management/)).
That *silent staleness* is precisely the drift US7 exists to kill — and the tools' answer is **suspect links**.

**The "suspect link" is the pattern to steal wholesale.** In Jama Connect, *"when upstream items change,
their downstream items are flagged as **suspect** so you can review any impact… the item might no longer be
correct or complete."* Crucially the flag is **never auto-resolved** — a human (or a configured rule)
**clears** the suspect link individually or in bulk after reviewing
([Jama: Suspect Tracking](https://www.jamasoftware.com/blog/2025/09/13/the-importance-of-suspect-tracking-in-requirements-management/);
[Jama: Clear suspect links](https://help.jamasoftware.com/ah/en/manage-content/coverage-and-traceability/relationships/clear-suspect-links.html)).
**Adopt this vocabulary directly:** a downstream write-back that *touches an element with recent human
edits or open human comments* is **suspect** and must be cleared by a human before it applies. Suspect
tracking is configurable *per field* — the granular lever Step 7 wants for its graduated-trust policy
(§Lens 6).

**Interchange-format lesson (ReqIF/DOORS):** the mature ecosystem moves requirement deltas between tools
via **ReqIF**, a vendor-neutral format that preserves *hierarchy and trace relationships*, with tools like
ReqXChanger maintaining bidirectional links between DOORS and downstream models
([Jama: ReqIF/DOORS](https://www.jamasoftware.com/blog/ibm-doors-software/);
[IBM: ReqXChanger bidirectional traceability](https://www.ibm.com/docs/en/engineering-lifecycle-management-suite/design-rhapsody/10.0.1?topic=developing-using-reqxchanger-bidirectional-traceability)).
The portable idea is not the heavyweight format — it's that **the link carries identity and relationship**,
so a change can be *traced and reviewed*, not blindly merged. Our `change_request.target_surrogate` +
`base_version_id` is the lightweight equivalent of a ReqIF trace link with a version stamp.

**Practitioner verdict:** the field that has done requirements round-trip longest does it as
*flag → human-review → clear*, never as silent two-way sync. Step 7 should ship the **suspect-flag +
gated-apply** model, not an auto-merge engine.

## Lens 2 — Tools & Technologies: reliable notification delivery (the dual-write trap) and the inbox surface

**FR-019 ("notify the user what changed + from where") hides a classic distributed-systems bug: the dual
write.** Step 7 must do two things when a downstream change lands — (a) persist the requirement change and
(b) emit a notification. If those are separate writes, a crash between them either loses the notification
(silent drift returns) or fires a notification for a change that rolled back (phantom). The industry-standard
fix is the **transactional outbox pattern**: write the business change *and* a row in an `outbox` table
**inside one DB transaction**, then a separate relay reads the outbox and delivers
([AWS Prescriptive Guidance: Transactional Outbox](https://docs.aws.amazon.com/prescriptive-guidance/latest/cloud-design-patterns/transactional-outbox.html);
[Outbox Pattern Explained — Streamkap](https://streamkap.com/resources-and-guides/outbox-pattern-explained)).

Key facts that shape our design:
- **The relay can poll the outbox or use Change Data Capture (CDC / Debezium reading the txn log).** For a
  single FastAPI+SQLite/Postgres app, a **polling relay** (a background task draining `notifications_outbox`)
  is the right weight — CDC is over-engineering at this scale
  ([Outbox — Conduktor](https://www.conduktor.io/glossary/outbox-pattern-for-reliable-event-publishing)).
- **Delivery is at-least-once, not exactly-once → notifications must be idempotent.** A notification carries
  the `change_request_id`; the UI dedupes on it. (Standard outbox caveat
  — [SeatGeek: Transactional Outbox in production](https://chairnerd.seatgeek.com/transactional-outbox-pattern/).)
- **Outbox carries *domain events*, not raw row diffs** — e.g. `RequirementChangedByDownstream`, with the
  provenance + change-summary payload, not a SQL changeset
  ([AWS: domain consistency in event-driven architectures](https://aws.amazon.com/blogs/mt/achieve-domain-consistency-in-event-driven-architectures/)).

**The user-facing notification *surface*: W3C Linked Data Notifications (LDN).** This is the natural
companion to the W3C Web Annotation Protocol that Step 4 already adopted as its comment contract. LDN is a
**W3C Recommendation**: any resource advertises a receiving endpoint (an **Inbox**); *senders* POST JSON-LD
notifications to it; *consumers* GET them and process application-specifically; the three roles are cleanly
separated and interoperate
([W3C LDN](https://www.w3.org/TR/ldn/); [LDN — Wikipedia](https://en.wikipedia.org/wiki/Linked_Data_Notifications)).
Why it fits Step 7 like a glove:
- **A spec can advertise its own Inbox.** A downstream agent (sender) POSTs *"I changed FR-014, here's the
  provenance"* to the spec's inbox; the human UI and any watching agent (consumers) read it. This is the
  **same-door, agent-native** notification channel FR-013 demands — no separate human-vs-agent path.
- **Targets can use a third-party inbox** — forward-compat for a future centralized Diecast notification
  service without changing the contract.
- It pairs structurally with Step 4's Web Annotation Protocol (both are W3C JSON-LD-over-HTTP resource
  protocols), keeping the whole comment+notification surface coherent.

**Stack reality check (from Step 2/Step 4 code notes):** cast-server is build-free FastAPI + Jinja + HTMX,
no npm. Both the outbox relay (a background task) and an LDN inbox (`GET`/`POST` JSON endpoints + an HTMX
notification badge/panel) sit natively on this stack. **No new infrastructure, no message broker required**
for v2 — a DB outbox table + a polling task + JSON endpoints is the whole mechanism. (A broker is the
scale-up path, not a v2 need.)

## Lens 3 — AI/ML Approaches: agents as the typical write-back source, and LLM drift-detection as the trigger

**FR-013 says downstream *agents* are the typical write-back source — and the AI industry has already built
this exact loop under a different name: "documentation drift detection."** Tools like ReadMe's AI Writer and
the `doc-drift` GitHub Action run an **LLM over each PR to detect when a code change makes the docs stale,
then propose the doc update for human approval**
([ReadMe: AI Writer detecting doc drift](https://readme.com/resources/ai-writer-detecting-doc-drift);
[doc-drift GitHub Action](https://github.com/jbrockSTL/doc-drift)). Map it onto Step 7: substitute "code PR"
→ "downstream phase output (plan/exploration/execution note)" and "docs" → "requirements artifact." The
proven shape is **LLM proposes, human gates** — *never* LLM silently rewrites. That is the AI-native
validation of §Lens 6's contrarian framing from an entirely independent domain.

**The "same door" contract (FR-013) is free if write-back is an API, not a code path.** This mirrors Step 4's
finding exactly: AI code-review bots post findings through the *same* review-comment REST endpoint a human
uses, emitting **structured JSON** that a script reliably posts
([Cloudflare: orchestrating AI code review](https://blog.cloudflare.com/ai-code-review/)). For Step 7: a
downstream agent calls `POST /api/specs/{slug}/change-requests` with `author_type=agent`, the *identical*
endpoint a human "suggest an edit" action calls. `author_type` is **data, not a branch** — the forcing
function that keeps FR-013 satisfied by construction (and the same first-principles rule Step 4 derived).

**Provenance is what makes an agent write-back *auditable and reversible* — non-negotiable for AI authors.**
Every agent-originated change must record `author_type=agent`, the agent name, the originating activity
(run id), and the source artifact — so a wrong agent edit can be traced and rolled back. Event-sourcing's
metadata discipline applies directly: *"events can be stored with metadata about who performed the action
and when… capturing not just what changed, but why and who initiated it (correlation IDs, user info,
business context)"*
([Event Sourcing — Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing);
[Event Sourcing 101 — DZone](https://dzone.com/articles/event-sourcing-guide-when-to-use-avoid-pitfalls)).

**Contrarian AI temptation — auto-merge non-conflicting agent changes — concede only a narrow lane.** The
seductive "AI-native" move is full auto-apply for any change that doesn't textually conflict. **Reject it
for modifications of human-authored elements** (audit/data-loss hazard). **Concede a narrow auto-apply lane
only for pure *additions*** — a brand-new FR the planner derived from a genuine discovery — delivered as
*"applied, FYI-notified"* rather than gated. Additions can't overwrite human intent (nothing to lose), so
the blast radius is bounded. This is the graduated-trust policy of §Lens 6, justified from the AI side.
**A second AI temptation — embedding/semantic re-anchoring of a change to "the requirement it's about" —
is rejected for the same reason Step 4 rejected it for comments: non-deterministic, un-auditable. Anchor to
the stable surrogate; use similarity at most as a *human-confirmed hint* when the target was deleted.**

## Lens 4 — Community & Open Source: the provenance standard (W3C PROV) and the living-documentation movement

**W3C PROV is the open standard purpose-built for "who changed what, from where, derived from what."** It is
a W3C Recommendation family (PROV-DM data model, PROV-O OWL ontology) describing *"a standardized form of
knowledge graph providing an audit trail of what a system performed… references to the people, datasets and
organisations involved; attribution of data; and data derivation"*
([W3C PROV-DM](https://www.w3.org/TR/prov-dm/);
[The W3C PROV family of specifications](https://dl.acm.org/doi/10.1145/2452376.2452478)). Its **three core
types map 1:1 onto FR-020**, and this mapping is the spine of Step 7's provenance design:

| PROV concept | Step 7 binding | FR-020 phrase |
|--------------|----------------|---------------|
| **Entity** | the requirement element revision (Step 2 surrogate + version) | "the change" |
| **Activity** | the downstream phase run (exploration / planning / execution agent run) | "which phase originated it" |
| **Agent** | the human owner or the AI agent (`cast-high-level-planner`, …) | "which agent originated it" |
| `wasGeneratedBy` (Entity→Activity) | this revision was produced by the planning run | provenance link |
| `wasAttributedTo` (Entity→Agent) | attributed to agent X / human Y | "which agent" |
| `wasDerivedFrom` (Entity→Entity) | derived from `plan.collab.md` / prior requirement version | "from where" |

W3C explicitly notes PROV's **URI-per-version scheme is "particularly suited for expressing provenance of
documents, since each URI denotes a specific version of a document"** — i.e. PROV was *designed* for exactly
our versioned-requirements case
([ericP's PROV-DM notes](https://www.w3.org/2012/10/prov-dm)). **Posture (same as Step 4 took with W3C Web
Annotation): borrow the *shape and vocabulary*, not the full JSON-LD ceremony.** Store
`(entity_rev, activity_run, agent, relation)` as plain columns/rows with PROV-aligned naming; a full PROV-O
export is a cheap forward-compat add-on, never a v2 dependency.

**The "living documentation" movement is the philosophical frame for *why* this matters.** Cyrille
Martraire's *Living Documentation* (and Gojko Adzic's *Specification by Example* before it) defines four
principles, the first of which is **Reliable — "all documentation is accurate and in sync with the software
being delivered at any point in time"** — and **Low-Effort — "minimize the work to keep it current even
under change"**
([InfoQ: DDD and Living Documentation](https://www.infoq.com/news/2015/06/ddd-living-documentation/);
[Leanpub: Living Documentation](https://leanpub.com/livingdocumentation)). The whole BDD/Cucumber insight is
that *a doc out of sync should announce itself loudly* ("whenever a test fails, it signals the documentation
is no longer in sync"). **Step 7's round-trip + suspect-flag + notification is precisely the mechanism that
makes Diecast's requirements "Reliable" in Martraire's sense** — without it, requirements are the static
README that *"silently goes stale the moment exploration finds a new constraint"* (the goal's own Multi-Lens
insight). FR-019's notification is the BDD "loud failure" applied to a prose artifact.

**Docs-as-code is the proven OSS workflow for keeping docs from drifting, and it independently confirms the
gate.** The community consensus: *"documentation is stored in Git, updated through pull requests, integrated
into CI/CD — because documentation follows the same lifecycle as the code, updates happen in sync rather than
as an afterthought"*
([Docs-as-Code: prevent API doc drift](https://blog.docuwiz.io/p/docs-as-code-how-to-prevent-api-documentation)).
The **PR** — a *reviewed, attributed, gated* change — is the OSS-native form of "propose + notify + gate."
Step 7's `change_request` is a PR for a requirement element.

## Lens 5 — Frameworks & Patterns: the concrete recommended design

Step 7 is a **thin composition layer** over Step 2 (stable surrogates + immutable versions = write target +
conflict unit) and Step 4 (structured diff + same-door API + append-only trail). The net-new pieces are a
`change_request` entity, a provenance record, a notification outbox, and a conflict predicate.

### Schema (extends Step 4's `spec_versions` / `spec_elements` / `comment_events`)

```
change_requests(
  id PK, goal_slug FK, target_surrogate? FK→spec_elements,   -- null ⇒ pure addition (no existing target)
  base_version_id FK→spec_versions,                          -- the version the change was DERIVED from (conflict anchor)
  kind ∈ {addition, modification, annotation},
  proposed_body,                                             -- the new/added text, or annotation note
  status ∈ {proposed, accepted, rejected, conflicted, superseded, applied},
  -- provenance (PROV, denormalized for v2; PROV-O export is forward-compat) --
  origin_phase ∈ {exploration, planning, execution, …},     -- PROV Activity class
  origin_activity_id,                                        -- run id of the downstream agent (PROV Activity)
  origin_artifact_path,                                      -- wasDerivedFrom source (e.g. plan.collab.md)
  author, author_type ∈ {human, agent},                     -- PROV Agent + wasAttributedTo
  created_at, decided_at?, decided_by?, decided_by_type?
)
provenance(                                                  -- optional explicit PROV triples; or fold into change_requests for v2
  id PK, entity_rev_id FK→spec_elements, activity_id, agent_id,
  relation ∈ {wasGeneratedBy, wasAttributedTo, wasDerivedFrom, used}, at
)
notifications_outbox(                                        -- transactional outbox (written in same txn as the change)
  id PK, goal_slug FK, change_request_id FK, payload JSON,   -- payload = change-summary delta + provenance
  status ∈ {pending, delivered}, created_at, delivered_at?
)
change_request_events(                                       -- append-only audit trail (Step 4's comment_events, generalized)
  id PK, change_request_id FK, event ∈ {proposed, accepted, rejected, conflicted, applied, superseded},
  actor, actor_type, at, note
)
```

### 1. Write-back path (FR-018): propose, don't commit

- A downstream phase calls **`POST /api/specs/{slug}/change-requests`** (the *same door*, FR-013) with
  `target_surrogate` (or null for an addition), `kind`, `proposed_body`, `base_version_id`, and provenance.
- **Pure additions** (a new FR derived from a discovery) → fast-track lane: may auto-apply as a new element
  appended to the current version, status `applied`, with an **FYI notification** (§Lens 3/6). Nothing is
  overwritten, so blast radius is bounded.
- **Modifications / annotations to existing elements** → status `proposed`; **never** mutate the element in
  place. This is US7 S1 verbatim: *"appended additions or annotated edits, not silent rewrites."* An
  annotation can render exactly like a Step 4 comment (reuse that surface) tagged `origin=downstream`.

### 2. Provenance capture (FR-020): PROV triple, surfaced in the change summary

- On apply, the resulting element revision records `wasGeneratedBy` (origin activity), `wasAttributedTo`
  (origin agent), `wasDerivedFrom` (origin artifact + prior revision) — §Lens 4's mapping.
- The write-back appears in the **next version's change summary using Step 4's structured element diff,
  with a provenance badge**: `+FR-021 — added by planning · agent cast-high-level-planner · derived from
  plan.collab.md`. **No new diff engine** — Step 4's `{added, removed, modified}` set arithmetic, plus a
  provenance column. This is the explicit Step 4→Step 7 hand-off.

### 3. Notification (FR-019): transactional outbox → LDN inbox

- In the **same transaction** that commits the change_request (or applies an addition), insert a
  `notifications_outbox` row whose `payload` is the change-summary delta + provenance. (Solves the dual-write
  trap, §Lens 2.)
- A background **polling relay** drains the outbox and delivers to the spec's **LDN Inbox** (and/or an HTMX
  notification badge/panel). At-least-once → UI dedupes on `change_request_id`.
- Payload answers both FR-019 questions in one shape: **what changed** (the element diff) + **from where**
  (the provenance). Humans see a notification; watching agents consume the same LDN resource.

### 4. Conflict detection + surfacing (US7 S4): three-way merge against `base_version_id`

- At apply time, run the **optimistic-concurrency predicate**: compare the target element's *current* body
  against its body *at `base_version_id`* (the version the change assumed).
  - **Unchanged since base** → clean apply (or clean addition).
  - **Diverged since base** (esp. edited by a human, or carrying open human comments) → status `conflicted`
    → **surfaced to the user, never overwritten.** This is `conflictDetection: VERSION`
    ([AWS AppSync: conflict detection/resolution](https://docs.aws.amazon.com/appsync/latest/devguide/conflict-detection-and-resolution.html))
    and git's 3-way merge with a common base
    ([Automatic Detection/Resolution of Merge Conflicts (arXiv)](https://arxiv.org/pdf/2102.11307)).
- **Conflict UX = reuse the review surface**, rendered as a special diff with `accept-incoming` /
  `keep-current` / `merge` actions (the 3-way choice), borrowing Jama's **suspect-link** semantics: the
  element is flagged suspect until a human (or a trusted-rule) **clears** it (§Lens 1). The conflict and its
  resolution are logged in `change_request_events` (append-only audit).

### 5. Agent-as-source contract (FR-013)

- One endpoint, `author_type` is data. A human "suggest edit" and an agent write-back are the same `POST`.
  The UI has **no privileged write path** the agent lacks — the first-principles forcing function (§Lens 7),
  identical to Step 4's comment API.

## Lens 6 — Contrarian View: kill "auto-sync," kill "needs a merge engine," kill "needs a new pipeline"

**Premise 1 (kill): "Living source of truth ⇒ automatic two-way sync."** Dead, and this is the load-bearing
contrarian point. The documentation-drift literature is unambiguous that *staleness* is the enemy — *"60% of
documentation becomes outdated within six months… silent, with no error logs or failing tests"*
([Document360: Documentation Drift](https://document360.com/blog/documentation-drift/);
[zivodoc: API Documentation Drift](https://zivodoc.com/blog/api-documentation-drift-prevention/)). The
*reflexive* fix is "automate the sync." **But for a *source-of-truth* document, silent auto-sync is worse
than drift** — it is untraceable overwrite of human intent (US7 S4's named failure). The resolution is not
auto-sync; it is **propose + notify + gate**: the write-back is a *suggestion* (Google-Docs **suggesting
mode**: insertions shown, *"each edit is marked as a suggestion… a checkmark to accept, incorporating the
change; reject reverts to the original"*
([Google Docs: Suggest edits](https://support.google.com/docs/answer/6033474))) or a *PR* (docs-as-code).
The owner stays in the loop precisely *because* truth must be **governed, not eventually-consistent.**

**Premise 2 (kill): "Round-trip needs a real merge/CRDT engine."** No. CRDT/OT exist for *concurrent
multi-cursor co-editing*, which Step 4 already put **out of scope** (single-writer, async). The residual
problem is *asynchronous* reconciliation against a known base — solved deterministically by **three-way merge
+ optimistic concurrency** (§Lens 5). Reaching for CRDT here is the same category error Step 4 diagnosed for
React: importing co-editing machinery for a non-co-editing feature.

**Premise 3 (kill): "Round-trip is a new pipeline to build."** No — it is a **composition** of primitives
that already exist after Steps 2 and 4: stable surrogates + immutable versions (the write target and the
conflict unit), the structured-diff change-summary (the FR-020 surface), the same-door API and append-only
trail (FR-013 + audit). Step 7's net-new is a *thin* layer: a `change_request` entity, a PROV triple, a
notification outbox, and one conflict predicate. Anyone scoping this as a from-scratch subsystem is
double-paying for Step 2/Step 4 work.

**The genuine risk to concede: gate fatigue.** If *every* trivial agent addition demands human sign-off,
owners will rubber-stamp (defeating the gate) or disable it (defeating the feature). **Mitigation: graduated
trust, keyed to blast radius, not a uniform gate** — exactly Jama's *per-field, configurable* suspect
tracking (§Lens 1):
- *Pure additions* (new FR derived from a discovery) → **auto-apply + FYI notification** (nothing
  overwritten).
- *Modifications / annotations* of existing **human-authored** elements → **always gate**.
- *Conflicts* (diverged from base) → **always surface**, never auto-resolve.
Tune the gate by what a change can *destroy*, not by who authored it.

## Lens 7 — First Principles: decompose to irreducible primitives

- **What is "living source of truth"?** Two atoms: **currency** (the artifact's state reflects the latest
  decisions) *and* **accountability** (every change is attributable and reversible). Currency without
  accountability = silent drift/data-loss. Accountability without currency = a stale audit log. US7 needs
  *both* — which is why notification (currency-forcing) and provenance (accountability) are co-equal, not
  one-or-the-other.
- **What is a write-back, atomically?** *A proposed state transition on an element, originating outside the
  requirements phase, carrying its origin and its base.* The two fields the naïve design forgets are exactly
  the two that make the feature work: **origin** (enables notification + audit) and **base** (enables
  conflict detection). Everything else (body, kind, status) is ordinary data.
- **What is provenance, minimally?** *(Entity, Activity, Agent)* — W3C PROV's three nuclei. You cannot
  express "who changed what, from where" with fewer than three. Drop the Agent → no attribution; drop the
  Activity → no "which phase"; drop the Entity → nothing to attribute. PROV is irreducible, not ceremonial.
- **What is a conflict, atomically?** *The element diverged from the base the change assumed.* Detectable
  **iff** every element carries a version identity (Step 2) *and* every change carries its base version.
  No version identity ⇒ no conflict detection ⇒ silent overwrite is the *only* possible behavior ⇒ US7 S4 is
  unsatisfiable. **Version identity is load-bearing for conflict surfacing, exactly as it was for diffs in
  Step 4.** This is the second time the same Step 2 primitive proves to be the keystone — strong evidence
  the architecture is coherent.
- **What does "agent is the typical source" require, minimally?** That the write-back API has **no
  human-privileged path** — `author_type` is data, not a code branch. If the human UI ever writes back
  through a path the agent can't call, FR-013 is already violated. (Same forcing function as Step 4's
  comment API.)
- **What does "notify without losing the notification" require, minimally?** That the change and its
  notification commit **atomically** — one transaction. Two transactions admit a crash window that
  reintroduces drift. (The outbox, §Lens 2.)

**First-principles conclusion:** Step 7's irreducible primitives are
*(stable element identity + immutable version)* [Step 2] +
*(structured diff + append-only trail + same-door API)* [Step 4] +
*(change_request{origin, base} + PROV triple + atomic notification)* [Step 7's net-new].
None is a frontend or realtime concern. Round-trip needs **governance, provenance, and an atomic
notification channel** — not CRDT, not React, not a broker, not a new pipeline.

## Open questions / hand-offs for the playbook & later steps

1. **Notification surface decision (overlaps Step 4 open-question #4 — coordinate).** Build a W3C-LDN-style
   inbox, or extend an existing cast-server notification UI? Recommend: a minimal LDN-aligned JSON inbox +
   an HTMX badge/panel for v2; full LDN federation later. Step 4 already flagged this overlap — resolve the
   two together so comments and round-trip notifications share one surface.
2. **Graduated-trust policy is an owner/product decision, not a default.** Which write-backs auto-apply
   (additions) vs gate (modifications) vs always-surface (conflicts) — and is auto-apply on at all in v2, or
   does v2 gate everything for safety and relax later? Recommend gate-everything-except-pure-additions for
   v2; expose the policy as configuration (Jama-style per-field) rather than hardcoding.
3. **v2 build boundary (mirror US6's seam-vs-pipeline split).** v2 should build the *receiving* mechanism —
   the `change-requests` endpoint, provenance capture, conflict predicate, notification outbox — and
   demonstrate it via a **simulated** downstream change (SC-006), even though real planning/execution agents
   aren't *emitting* write-backs yet (wiring non-requirements phases is out of scope, per US6/Constraints).
   Confirm this scoping with the owner: *build the inbound contract + conflict + notification; defer the
   outbound emitters.*
4. **PROV depth.** Recommend denormalized PROV-aligned columns on `change_requests`/`spec_elements` for v2,
   with PROV-O/JSON-LD export as forward-compat (same posture Step 4 took with W3C Web Annotation). Confirm
   no immediate interop requirement forces full PROV-O now.
5. **Conflict-resolution UX granularity.** Three-way `accept-incoming / keep-current / merge` reusing the
   review/comment surface — but "merge" on prose elements may need a manual edit box rather than an auto
   3-way text merge. Recommend: surface both versions + a free edit field; don't build an auto-textual-merge
   for v2 (element granularity from Step 2/Step 4 keeps conflicts small).
6. **Element granularity dependency (shared with Step 4).** Conflict precision = element granularity. If
   Step 2 lands coarse elements, a downstream change to one clause conflicts the whole element. Reinforces
   Step 4's request for scenario/clause-level surrogates.
